200 Days, 2000 Problems, Whatever Happens Happens

Look, I'll be straight with you. I'm doing something that's probably stupid: solving 10 LeetCode problems every single day for the next 200 days. That's 2000 problems total, plus learning JavaScript frameworks, plus building projects that don't suck.

Am I documenting this because I think I'm special? No. Am I doing it for accountability? Maybe. Mostly I'm doing it because I'm curious what happens when you actually stick to something this specific for this long.

What's the Deal

The plan is simple:

  • 10 LeetCode problems every day (no exceptions, no excuses)
  • Get actually good at JavaScript and React ecosystem
  • Build some projects that combine my AI background with web dev
  • See if I can position myself better than 70% of developers out there

Why I'm Not Starting From Zero

I'm not one of those "I learned to code in 3 months" stories. I've already got Python/Flask down, built a few full-stack apps, and I'm studying AI/Data Science. I've even led teams at hackathons and won some stuff.

But here's the thing — having some skills and being consistently good at problem-solving are different beasts. I can build things, sure. Can I solve any algorithmic problem thrown at me? Can I architect systems that don't fall apart? Can I code in JavaScript without googling basic syntax?

Honestly? Probably not yet.

The Only Non-Negotiable Thing

10 problems a day. Every day. For 200 days.

5 in the morning before anything else happens. 5 at night before I call it a day. Sick, tired, busy, celebrating, hungover — doesn't matter. This number doesn't change.

Rough Game Plan

I'm breaking this into chunks because 200 days feels abstract:

Days 1-70: Get the Foundation Right

Focus on arrays, strings, basic data structures. Learn TypeScript properly this time. Stop writing JavaScript like it's Python.

Days 71-140: The Hard Stuff

Trees, graphs, dynamic programming. Next.js and system design. Build something that actually integrates AI in a useful way.

Days 141-200: Polish and Hunt

Company-specific problem sets. Advanced algorithms. Portfolio that doesn't look like a tutorial project. Get ready to actually apply places.

What Makes This Different

I'm not trying to become a developer — I already am one. I'm trying to become the kind of developer who can solve problems quickly, build systems that scale, and combine AI with web development in ways that actually matter.

Also, I'm not pretending this is going to be easy or that I'll stick to every detail of the plan. Things will go wrong. I'll have bad days. Some problems will destroy me. Some projects will fail.

That's fine. The point isn't perfection — it's consistency.

What You'll Get Here

I'll probably update this thing regularly with:

  • Which problems made me want to quit programming forever
  • JavaScript concepts that finally clicked
  • Projects that turned out better (or worse) than expected
  • The inevitable weeks where everything goes off track

The Real Question

Can someone actually do 2000 LeetCode problems in 200 days while learning frameworks and building projects? I don't know. Most people would say it's too much.

But here's what I do know: I've never really tested my limits. I've never committed to something this specific for this long. I've never had a plan this detailed.

So we'll see what happens.

Day 1 starts tomorrow. Either I'll have an interesting story to tell in 200 days, or I'll have learned something about why most people don't finish what they start.

Both outcomes seem worth finding out.
